# AI OCR 跨供应商兜底设计

## 背景

生产任务 `d9d4d0c25ff94b218949b6f750aab7cb` 的 9 个单图批次中有 2 个失败：智谱和 SiliconFlow 各有一个请求超过应用设置的 90 秒读取超时。任务内存峰值约 118 MB，远低于 Render 的 512 MB 上限，因此本次失败不是 OOM。

当前严格完整性检查会在任一图片批次失败时停止任务，不返回或展示部分结果；这一约束保持不变。

## 选定方案

仅将 `AI_OCR_PROVIDER_FALLBACK` 从 `false` 改为 `true`，其余生产参数保持：

- `AI_OCR_IMAGE_BATCH_SIZE=1`
- `AI_OCR_IMAGE_WORKERS=3`
- `AI_OCR_DUAL_PROVIDER=true`
- `AI_OCR_TIMEOUT_SECONDS=90`
- `AI_OCR_LOCAL_FALLBACK=false`

每个图片批次仍按轮转方式首先分配给一家供应商。只有首选供应商失败时，当前批次才改用另一家供应商重试一次。正常批次不增加请求，三个工作槽的总并发上限也不改变。

未采用的方案包括：提高到 4 并发、单独提高超时阈值、启用本地 RapidOCR。它们分别会增加供应商排队风险、延长卡死请求占用工作槽的时间，或显著增加内存风险。

## 完整结果和错误处理

重试成功的批次与其他批次一起进入事件合并和后续 AI 匹配。只有全部图片识别批次和全部 AI 匹配批次完成后，前端才一次性展示最终结果。

若同一图片在两家供应商上都失败，任务继续抛出 `AiOcrIncompleteError`，不返回 7/9 之类的部分结果。失败信息应包含两家供应商的错误，便于后续诊断。

## 配置、测试和部署

仓库中的 `render.yaml`、`.env.example` 和 README 示例应同步为开启跨供应商兜底，避免生产配置与文档漂移。配置测试应先证明当前 `false` 不满足目标，再改为 `true` 并通过。

完整测试通过后，提交并推送配置变更，更新现有 Render 服务的生产环境变量并触发部署。部署验证包括：最新部署状态为 `live`、首页和状态接口返回成功、没有新增错误日志，并复查内存仍显著低于 512 MB。

## 回滚

如跨供应商重试造成不可接受的调用量或尾延迟，将 `AI_OCR_PROVIDER_FALLBACK` 恢复为 `false` 并重新部署；其他并发、批次和超时参数无需变化。

## 成功标准

- 正常路径仍以最多 3 个单图任务并行运行。
- 单一供应商超时时，同一图片能由另一供应商接管。
- 任何情况下都不向用户展示部分 AI 分析结果。
- Render 内存不超过 512 MB，部署后无新增服务错误。
